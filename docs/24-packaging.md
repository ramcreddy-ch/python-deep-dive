# 24. Packaging, Dependencies & Virtual Environments ŌĆö Production Deep Dive

> "It works on my machine" is solved by standardizing packaging. Managing Python dependencies in production has historically been chaotic. Mastering `virtualenvs`, `pip`, and modern resolvers (`poetry`) is the difference between a resilient ML platform and a brittle set of scripts that break on every minor version upgrade.

---

## ­¤öŹ The Basics: Venvs & Pip

### Why Virtual Environments Exist
Python installs packages globally (in system directories like `/usr/lib/python3.10/site-packages`) by default. If Script A requires `pandas==1.0` and Script B requires `pandas==2.0`, conflicts arise instantly.

Virtual Environments (`venv`) isolate the execution environment. They create a dedicated local folder structure containing a distinct Python binary and a distinct `site-packages` folder.

```bash
# SRE Standard Workflow
python3 -m venv .venv
source .venv/bin/activate
# Now, pip strictly writes to the .venv directory, ignoring the OS entirely
pip install fastapi
```

### The Pitfalls of `requirements.txt`
`pip freeze > requirements.txt` is an amateur pattern for applications. It captures sub-dependencies dynamically based on whoever dumped it. If you deploy this across Linux and Windows, OS-specific compiled dependencies will crash on install.

*   `requirements.in`: Contains high-level human requirements (e.g., `fastapi`, `uvicorn`).
*   `requirements.txt`: Should only be generated by a tool like `pip-tools` resulting in deterministic pinned hashes.

## ­¤ÅŁ Modern Dependency Management (Poetry/UV)

The industry has moved beyond `setuptools` and `setup.py` towards declarative `pyproject.toml` standards (defined by PEP 518).

### Poetry (The Standard)
Poetry abstracts environment creation, dependency resolution, and packaging. It separates `dependencies` (required for production) from `dev-dependencies` (testing/linting frameworks).

```toml
# pyproject.toml
[tool.poetry]
name = "ml-platform-sdk"
version = "1.2.0"
description = "Internal tooling for MLOps."

[tool.poetry.dependencies]
python = "^3.10"
boto3 = ">=1.26"
fastapi = "*"

[tool.poetry.group.dev.dependencies]
pytest = "^7.4.0"
black = "*"
```

### The Lockfile (`poetry.lock`)
When you run `poetry install`, Poetry solves the complex graph of requirement tree limits and writes the absolute, exact versions securely to `poetry.lock`. This file is committed to Git. When the CI/CD pipeline runs `poetry install --no-dev`, it guarantees identical byte-for-byte environments globally.

## ­¤ÜĆ Building & Publishing Private Packages

In enterprise contexts, you don't copy-paste utility functions across 10 repositories. You isolate them in an internal package and publish to a private PyPI server (like AWS CodeArtifact or Nexus).

```bash
# Building the wheel (.whl) and source distribution (.tar.gz)
poetry build

# Configuring poetry to hit a private artifactory
poetry config repositories.internal https://nexus.corp.local/repository/pypi-internal/

# Publishing artifacts
poetry publish -r internal -u $NEXUS_USER -p $NEXUS_PASS
```

### Understanding Wheels (`.whl`)
A wheel is a pre-compiled binary package standard. When you `pip install` a wheel containing C++ code (like PyTorch or Pandas), the package installs instantly because it skips executing `gcc` to compile the C source files on the host machine. If you distribute packages containing C-extensions, you must compile wheels for the specific OS architectures you target.

## ­¤ż¢ MLOps Application: Conda vs Pip

Python manages *Python* dependencies. If your deep learning models require hardware-level Cuda Toolkits, specific C++ Protobuf binaries, and optimized OpenBLAS libraries, Pip is mostly blind to them.

**Conda** is a cross-language, OS-level package manager. It provisions virtual environments containing both Python modules *and* native C/C++ libraries.

```yaml
# environment.yml (The MLOps equivalent to pyproject.toml)
name: tf-gpu-env
channels:
  - conda-forge
  - nvidia
dependencies:
  - python=3.10
  - cudatoolkit=11.8
  - tensorflow-gpu=2.12
  - pip:
      - mlflow==2.1
```

## ­¤Ä» Senior Engineer Interview Questions

**Q1: We deployed our `requirements.txt` to the AWS Lambda Python 3.10 runtime. It contains `cryptography`. On developer Macs it works flawlessly; on Lambda, it fails with an `ImportError` citing missing `.so` files. Why?**
> **Answer:** Packages like `cryptography` require compiling native C code against host OS headers during install via `pip` (unless pre-compiled wheels map perfectly). A developer's Mac running ARM processes compiles Mac binaries. When these dependencies are zipped and uploaded raw to a Linux x86 Lambda runtime, the OS cannot read the foreign compiled binaries. The fix is compiling the deployment zip inside a Docker container mimicking the target architecture (`amazonlinux:2`), creating compatible Linux `.whl` files natively.

**Q2: What is the Python `sys.path` and how does the interpreter actually locate imported modules?**
> **Answer:** `sys.path` is a list of strings representing directory paths. When you type `import utils`, Python searches these directories sequentially. The order matters radically: 1. The local directory of the script executing, 2. `PYTHONPATH` environment variables, 3. The standard library internals, 4. The `site-packages` directory of the active virtual environment. If a hacker or accident places a malicious `math.py` in the local execution directory, Python imports it instead of the builtin `math` module (Shadowing).

**Q3: Describe the difference between placing a dependency in `install_requires` versus `extras_require` inside a traditional `setup.py`.**
> **Answer:** `install_requires` defines hard dependencies. If the package is pip-installed, these downstream packages trigger automatically and inevitably block execution if missing. `extras_require` defines optional capability sub-trees. For example, `pip install "my_platform[aws]"` triggers the `aws` extra key to dynamically pull down heavy `boto3` binaries, whereas a user who only wanted local functionality skips the bloat and just executes `pip install my_platform`.

**Q4: A security team audits your Container build process. Your Dockerfile executes `RUN pip install -r req.txt`. The security team mandates that building Docker images must be 100% deterministic, meaning identical source code must always produce mathematically identical Docker layers across time. Why does your script fail this mandate?**
> **Answer:** If `req.txt` states `flask>=2.0`, pip solves the graph dynamically based on what happens to exist on PyPI at the exact millisecond of the build. If Flask 3.0 gets published overnight, tomorrow's Docker build implicitly pulls down the new major version which might contain breaking changes. The build succeeds, but the behavior changes unpredictably. The fix is generating and utilizing a highly pinned Lockfile (like `requirements.txt` outputs from `pip-compile` containing sha256 artifact hashes, or `poetry.lock`), ensuring dependencies remain identically locked until manual updates are authorized.
